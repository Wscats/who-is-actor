# Who Is Actor — Git 仓库开发者画像技能

> 🔗 **项目地址：** [https://github.com/wscats/who-is-actor](https://github.com/wscats/who-is-actor)

零*安装*依赖，零脚本。纯粹通过原生**只读** `git` 命令和标准 Unix 文本工具（`cut`、`sort`、`awk`、`grep` 等 — 大多数系统已预装）采集数据，AI 仅负责对**已在本地完成聚合与脱敏的统计指标**做解读，输出协作模式报告。

> ⚠️ **使用前必读 — 隐私、知情同意与适用范围**
>
> 1. **可能涉及他人数据。** Git 仓库中通常包含运行本技能的用户**之外**他人的提交元数据（显示名、时间戳、提交消息主题、文件路径）。运行本技能前，用户必须确认自己有权分析该仓库，并且这样做不会违反工作场所、合同或当地法律义务。在团队场景下，代理**应当**提醒用户提前告知被分析的贡献者。
> 2. **不得用于人事/绩效决策。** 本技能输出（包括"参与度指数"和任何个人维度评分）**严禁**作为绩效考核、招聘/解雇、薪酬调整、裁员、排名或对个人作出人事性判断的依据。代理必须拒绝此类请求，仅以非个人化、聚合性的角度回应。
> 3. **提交元数据可能包含敏感内容。** 提交消息和文件名可能包含工单号、事故引用、密钥、客户名称、URL 或其他机密信息。本技能**强制要求**：此类原始文本仅在本地处理，只有经过聚合与脱敏的指标才能进入 AI 模型上下文。具体绑定规则见"敏感数据过滤规则"。
> 4. **只读、限定范围、无网络。** 本技能仅执行命令白名单中列出的只读 git 子命令，仅针对用户提供的单一仓库路径运行。不写入、不联网、不越出仓库根目录。

> **"零依赖"说明：** 本技能不安装任何东西 — 没有 pip 包、没有 npm 模块、没有自定义脚本。但它**确实需要**以下标准系统工具已在主机上可用：`git`、`cut`、`sort`、`uniq`、`awk`、`grep`、`sed`、`wc`、`head`。这些在几乎所有类 Unix 系统（macOS、Linux）上都已预装。Windows 用户请使用 Git Bash 或 WSL。

## 安全规范

> **所有 shell 命令参数在执行前必须经过严格校验，以防止命令注入攻击。**

### 预览模式 / Dry-Run（推荐首次使用）

在执行任何命令之前，代理**应当**提供 **预览模式（dry-run）**：

1. 收集并校验所有参数
2. 构建将要执行的完整命令列表
3. **将所有命令展示给用户审查，不实际执行任何命令**
4. 等待用户明确批准后再实际执行

触发预览模式的说法：
```
💬 "先列出要执行的命令，不要运行"
💬 "做一次 dry run"
💬 "Show me the commands first before running them"
```

> 这使用户可以验证每条命令是否严格匹配下方的白名单。

### 命令白名单（仅允许以下命令）

本技能 **仅允许执行以下预定义的只读 git 子命令**。不得执行任何其他 shell 命令：

| 允许的命令 | 用途 | 是否修改仓库？ |
|-----------|------|---------------|
| `git -C <path> rev-parse --is-inside-work-tree` | 验证路径是否为有效 Git 仓库 | ❌ 只读 |
| `git -C <path> rev-parse --show-toplevel` | 当用户提供的路径是仓库子目录时，解析出仓库根目录 | ❌ 只读 |
| `git -C <path> shortlog -sn --all` | 获取贡献者列表及提交次数 | ❌ 只读 |
| `git -C <path> log ...` | 获取提交历史详情（仅限只读参数） | ❌ 只读 |
| `git -C <path> diff --stat ...` | 获取变更统计 | ❌ 只读 |

> 任何未在上表中明确列出的 git 调用都必须被拒绝执行，即使看起来是只读的。向白名单添加新命令是对本技能安全约定的有意修改，必须同时更新此表与 dry-run 验证清单。

**严格禁止的命令类型：**
- ❌ 任何写入操作：`git push`、`git commit`、`git merge`、`git rebase`、`git reset`、`git checkout`、`git branch -d`
- ❌ 任何非 git 命令：`curl`、`wget`、`python`、`node`、`bash -c`、`sh`、`eval`、`rm`、`cp`、`mv`
- ❌ 任何文件写入或重定向：`>`、`>>`、`tee`（管道 `|` 仅允许连接只读文本处理工具：`cut`、`sort`、`uniq`、`awk`、`grep`、`wc`、`sed`、`head`）
- ❌ 任何网络操作：`git fetch`、`git pull`、`git clone`、`git remote`

> **如果 AI 代理尝试执行白名单之外的命令，用户应立即拒绝执行。**

### 输入校验规则（在执行任何 Git 命令之前必须完成）

1. **`repo_path`（仓库路径）校验：**
   - 必须是绝对路径（以 `/` 开头）
   - 不得包含以下危险字符或子串：`;`、`|`、`&`、`$`、`` ` ``、`(`、`)`、`>`、`<`、`\n`、`\r`、`$()`、`..`
   - 路径必须是真实存在的 Git 仓库（通过 `git -C <path> rev-parse --is-inside-work-tree` 返回 `true` 验证）
   - 如果校验失败，**立即中止并向用户报告错误 — 不得执行后续任何命令**

2. **`author`（作者名）校验：**
   - 仅允许字符：字母（a-z A-Z）、数字（0-9）、空格、连字符（`-`）、下划线（`_`）、点号（`.`）
   - **不允许 `@` 符号**（禁止邮箱格式以符合隐私保护规则）
   - 正则白名单：`^[a-zA-Z0-9 _.-]+$`
   - 最大长度：128 个字符
   - 如果输入包含 `@`，提示用户使用作者显示名，然后跳过该作者

3. **`since` / `until`（日期参数）校验：**
   - 必须匹配 ISO 日期格式：`^[0-9]{4}-[0-9]{2}-[0-9]{2}$`
   - 如果校验失败，忽略该参数并警告用户

4. **`branch`（分支名）校验：**
   - 仅允许字符：字母、数字、`/`、`-`、`_`、`.`
   - 正则白名单：`^[a-zA-Z0-9/_.-]+$`
   - 不得包含 `..` 子串
   - 如果校验失败，使用默认分支并警告用户

### 隐私保护规则

- **不采集开发者邮箱地址。** 所有 git 命令仅使用 `%an`（作者名）标识开发者，绝不使用 `%ae`（作者邮箱）。
- **`git shortlog` 使用 `-sn` 而非 `-sne`**，避免泄露邮箱地址。
- **`authors` 参数仅接受显示名，不接受邮箱地址。** 输入校验会拒绝包含 `@` 的值。
- **关于 `git --author` 语义的精确说明：** Git 内部会同时尝试将传入值与作者姓名、作者邮箱两个字段进行匹配。本技能**不**在 Git 层面禁用该匹配，而是在**输入侧**执行保证：`authors` 参数一旦包含 `@` 即被拒绝，且必须符合 `^[a-zA-Z0-9 _.-]+$`。实践上这意味着用户提供的值不可能是邮箱形式，因此只会有意义地匹配姓名字段；但这是关于**我们接受什么输入**的保证，而不是 Git 层面关闭邮箱匹配的开关。代理**不得**将邮箱匹配描述为“已禁用”。
- **邮箱地址绝不能出现于最终报告、中间提示、工具参数或任何送往 AI 的上下文**。
- **提交主题（`%s`）与 `--name-only` 产出的完整文件路径属于本地数据。** 部分白名单 git 命令会触及这类原始文本——例如 `git log ... --name-only`（产出完整路径），以及把 `%s` 立即在同一条管道中被 `awk '{print length}'` 等本地工具消费的命令（用于消息长度/关键词计数）。代理必须在本地把它们归约为数值聚合并立即丢弃；原始主题与完整路径**不得**进入任何 AI 提示或工具参数。**严禁**构造任何同时输出 `%s` 与逐提交结构化字段（哈希、numstat、文件名等）的命令，以避免原始主题与结构化数据被一并捕获。除非"敏感数据过滤规则"中明确的用户主动例外适用，否则不得突破上述边界。
### 敏感数据过滤规则（强制执行）

> **"本地数据"与"模型可见数据"的定义：**
>
> - **本地数据（local-only）**：白名单 git 命令的原始输出，包括提交主题（`%s`）以及 `--name-only` 产生的文件路径。它仅在代理的本地执行环境中读取，**唯一目的是计算聚合指标**，使用后**必须立即丢弃**。
> - **模型可见数据（model-bound）**：可以出现在发送给 AI 模型的任何提示中的**唯一**数据类别。它**只**包含数值计数、平均值、比率、百分比、文件扩展名直方图、小时/星期直方图，以及桶化的枚举（如 `bug_fix_commits=12`）。
>
> 原始提交消息文本、完整提交主题、完整文件路径、含自由文本的分支名，以及任何从提交历史派生的未脱敏字符串，**严格属于本地数据**。代理**不得**将此类字符串放入任何 AI 提示、系统消息、工具参数或其他离开本地的上下文中。

在将**任何**数据发送给 AI 模型进行分析之前，代理必须执行以下过滤流水线。每个步骤都是强制且不可跳过的：

1. **提交消息 — 仅本地处理：**
   - 代理运行白名单中产生 `%s` 的 `git log` 命令，并立即通过本地 Unix 工具（`awk '{print length}'`、`grep -cE`、`wc -l` 等）将其管道处理为：例如，平均长度、关键词计数（`fix`、`feat`、`revert`）、约定式提交合规率。
   - 中间的原始文本**不得**保留在任何变量、对话缓冲或会发送给 AI 模型的提示中。**只有最终的数值聚合结果**会被转发。
   - **默认行为：** AI 模型不接收任何提交消息文本，无论是完整的还是部分的。
   - **用户主动要求时的例外：** 当且仅当用户明确要求查看具体的提交消息时，代理必须按以下顺序：
     1. 对每条候选消息应用步骤 2 中所有脱敏规则；
     2. 将脱敏后的每条消息截断至最多 120 字符；
     3. 仅在**最终用户报告**中渲染脱敏、截断后的消息，**绝不**在中间 AI 提示或工具调用中出现；
     4. 警告用户：自动模式无法捕获所有敏感上下文，提交消息仍可能包含敏感信息。

2. **常见密钥模式自动脱敏（在任何字符串跨越本地边界之前应用，例如在用户报告中显示之前）：**
   - API 密钥 / 令牌：匹配 `(?i)(api[_-]?key|token|secret|password|credential|auth)[=:]\s*\S+` 的字符串
   - AWS 密钥：`AKIA[0-9A-Z]{16}`
   - 私钥：`-----BEGIN .* PRIVATE KEY-----`
   - 连接字符串：`(?i)(mysql|postgres|mongodb|redis)://\S+`
   - 通用密钥：在键值对模式中 `=` 或 `:` 之后出现的超过 20 字符的纯字母数字字符串
   - 匹配的内容替换为 `[已脱敏]`。

3. **文件名过滤 — 默认仅扩展名：**
   - 白名单中的 `git log ... --name-only` 命令**仅允许**在其输出**立即在本地被归约**的前提下使用，例如通过 `grep -oE '\.[^./]+$' | sort | uniq -c` 立刻收敛为扩展名直方图。`--name-only` 产生的完整文件路径属于本地数据，**不得**被发送给 AI 模型。
   - 对于返工检测（在概念上需要按文件分组），代理**必须**先在本地为每个路径生成稳定的不透明 ID（例如 `file_<sha1[:8]>`），只有返工次数与不透明 ID 才允许进入模型可见数据。
   - **用户主动要求时的例外：** 如果用户明确要求做文件级分析，代理在发送任何路径之前必须：
     1. 丢弃匹配 `.env`、`.credentials`、`*secret*`、`*password*`、`*token*`、`*.pem` / `*.key` 的路径段；
     2. 对剩余路径应用步骤 2 的正则脱敏；
     3. 警告用户：文件路径可能暴露内部项目结构与客户信息。

4. **作者显示名：**
   - 作者名仅作为聚合指标的不透明归属标签转发给 AI 模型。代理**不得**要求模型从名字本身推断个性、绩效或人格价值，**也不得**在任何提示中将作者名与原始提交主题耦合在一起。

### 仓库路径作用域规则

- 代理只能访问用户提供的特定仓库路径。
- 代理不得遍历父目录（`..`）或访问仓库根目录以外的文件。
- 代理不得列出或读取文件系统中的任意文件 — 仅允许针对已校验仓库的、白名单中的 `git` 命令。
- 如果用户提供的路径是仓库内的子目录，代理**必须**通过白名单命令 `git -C <path> rev-parse --show-toplevel` 解析仓库根目录，告知用户解析后的根路径，并在征得用户确认后再继续。

### 执行验证协议

由于本技能是纯指令型（无可执行代码），安全保证依赖于 AI 代理正确实现上述规则。**用户应在信任本技能处理敏感仓库之前验证执行情况。**

**验证步骤（先在安全的测试仓库上运行）：**

1. **Dry-run 测试：** 让代理使用预览模式分析测试仓库。验证：
   - 所有提议的命令都出现在上方的命令白名单表中
   - 没有命令使用 `%ae`（邮箱格式）或 `-sne` 标志
   - 所有用户提供的值（路径、作者、日期）都被正确引用

2. **输入校验测试：** 故意提供无效输入并验证拒绝：
   ```
   "分析 /tmp/test; rm -rf /"          → 代理必须拒绝（危险字符）
   "分析作者 user@email.com"           → 代理必须拒绝（不允许 @）
   "分析 since 2024-13-99"             → 代理必须拒绝或警告（无效日期）
   "分析分支 ../../etc/passwd"          → 代理必须拒绝（不允许 ..）
   ```

3. **数据过滤测试：** dry-run 之后，询问代理：
   ```
   "你会把什么数据发送给 AI 模型？"
   ```
   代理应确认只发送聚合指标（计数、平均值、百分比），而非原始提交消息或完整文件路径。

4. **脱敏测试：** 如果请求了提交消息，验证：
   - 消息被截断至 ≤120 个字符
   - 类似 `API_KEY=xxx` 的模式显示为 `[已脱敏]`
   - 消息仅出现在最终报告中，而非中间处理过程中

> **如果任何验证步骤失败，请勿在敏感仓库上使用本技能。** 请向技能维护者报告问题。

## 适用场景

- 当用户需要从仓库级、聚合视角查看提交节奏、流失率、返工信号等协作流程改进点时
- 当用户希望比较**团队整体**模式（而非个人），例如约定式提交合规率、周末/深夜比例、巴士因子风险时
- 当用户希望了解仓库的可见参与度分布，**作为对话的起点**（而非对个人下结论）
- 当用户需要一份结构化、数据驱动的输出，用于工作流程回顾讨论时

### 不适用场景（代理必须拒绝）

- 绩效考核、绩效校准、排名、招聘、解雇、裁员、薪酬调整或任何 HR 行为
- 对个人价值、智力或投入度作出排名或评判
- 在被分析人不知情或未同意的情况下对特定员工进行监视
- 分析用户尚未确认其有权检查的仓库

## 核心原则

> **不安装任何东西，不运行任何脚本。** 所有数据采集完全通过原生 git 命令（`git log`、`git shortlog`、`git diff --stat` 等）完成。AI 负责解读和评估。

> **安全第一。** 所有用户输入在被纳入 shell 命令之前，必须通过上述校验规则。任何校验失败必须导致终止或优雅降级 — 绝不跳过校验。

## 工作流程

### 步骤 1：确认分析参数

与用户确认以下参数（未指定则使用默认值）：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| **仓库路径** | 目标 Git 仓库的绝对路径 | （必填） |
| **目标作者** | 要分析的特定开发者，留空则分析全部 | 全部贡献者 |
| **日期范围** | 起止日期，ISO 格式 | 完整仓库历史 |
| **分支** | 要分析的目标分支 | 当前活跃分支 |

> **⚠️ 执行步骤 2 之前，所有参数必须按照上述"安全规范"进行校验。校验失败的参数不得用于命令构建。**

### 步骤 2：数据采集（纯 Git 命令）

按顺序执行以下 git 命令采集原始数据。**所有命令针对目标仓库目录运行 — 无需安装任何依赖。**

> 以下示例中的 `<repo_path>`、`<author>` 等为步骤 1 中经过校验的安全值占位符。

> 🔐 **本地数据边界提醒。** 本节中的每条命令都会产生原始文本（提交主题、文件路径等），按"敏感数据过滤规则"被归类为**本地数据**。下方命令中的管道（`| awk '{ print length }'`、`| grep -oE '\.[^./]+$'`、`| wc -l` 等）是**强制性的**：它们的职责是把上游原始文本**在进入 AI 模型之前**收敛为聚合数值。代理**不得**将上游原始文本捕获进任何会被发送给模型的变量、提示或工具参数。如果某一步骤的自然输出仍然包含原始文本（例如下面的返工检测日志），代理必须先在本地通过哈希、桶化或匿名化处理，再做进一步使用。

#### 2.1 贡献者总览

```bash
# 列出所有贡献者及提交次数（不包含邮箱以保护隐私）
git -C <repo_path> shortlog -sn --all
```

#### 2.2 逐作者提交详情

针对每位待分析的作者，执行以下命令（如指定了日期范围或分支，追加 `--since`、`--until`、`<branch>` 参数）：

```bash
# 逐提交元数据，用于节奏/规模分析（不包含提交主题 — 隐私原因）：
# 哈希、作者名、ISO 日期，以及逐提交的 numstat（增/删/路径）。
# 重要：%s（提交主题）**被有意从本命令中移除**，以避免原始主题与结构化
# 逐提交数据同时进入代理内存。%an（作者名）与逐文件 numstat 行是本地
# 中间值，必须在向 AI 模型传递前被归约为聚合计数/总和。--numstat
# 产出的文件路径遵从与 --name-only 同样的路径脱敏规则（见“敏感数据
# 过滤规则”§3）。
git -C <repo_path> log --author="<author>" --pretty=format:"%H|%an|%aI" --numstat

# 按小时统计提交次数（用于工作习惯分析）
git -C <repo_path> log --author="<author>" --pretty=format:"%aI" | cut -c12-13 | sort | uniq -c | sort -rn

# 按星期几统计提交次数（1=周一，7=周日）
git -C <repo_path> log --author="<author>" --pretty=format:"%ad" --date=format:"%u" | sort | uniq -c | sort -rn

# 新增/删除行数汇总
git -C <repo_path> log --author="<author>" --pretty=tformat: --numstat | awk '{ add += $1; subs += $2 } END { printf "added: %s, deleted: %s\n", add, subs }'

# 提交消息长度分布（本地只管道）。
# 重要：%s 在管道左侧产出原始文本，必须被右侧 awk '{print length}'
# 立即消费。代理**不得**拆解此管道、保留左侧输出、或在下游复用原始
# %s 文本；只有最终的逐提交长度整数可以被聚合与转发。
git -C <repo_path> log --author="<author>" --pretty=format:"%s" | awk '{ print length }'

# 涉及的文件类型
git -C <repo_path> log --author="<author>" --pretty=tformat: --name-only | grep -oE '\.[^./]+$' | sort | uniq -c | sort -rn | head -20

# 每日提交次数（用于频率分析）
git -C <repo_path> log --author="<author>" --pretty=format:"%ad" --date=short | sort | uniq -c | sort -rn | head -20

# 近期返工检测：7 天窗口内同一文件的修改频次。
# 重要：下方命令的原始输出包含提交主题（%s）和完整文件路径，属于本地数据。
# 代理必须在本地把它归约为：(a) 每位作者的返工计数、(b) 扩展名级别直方图，
# 然后才能让任何值进入模型提示。如确需保留按文件分组，请用不透明 ID 替换
# 路径（例如 file_<sha1[:8]>）。
git -C <repo_path> log --author="<author>" --pretty=format:"%ad" --date=short --name-only | head -200
```

> 注意：上面的返工检测命令相比朴素实现**有意丢弃了 `%s`（提交主题）**，因为提交主题必须保留为本地数据，而返工指标本身（同一文件在滑动窗口内被触碰的次数）并不需要主题。

#### 2.3 代码质量信号

```bash
# Bug 修复提交（消息包含 fix/bug/hotfix/patch）
git -C <repo_path> log --author="<author>" --grep="fix\|bug\|hotfix\|patch" --oneline -i | wc -l

# Revert 提交
git -C <repo_path> log --author="<author>" --grep="revert" --oneline -i | wc -l

# 大提交（>500 行变更）
git -C <repo_path> log --author="<author>" --pretty=format:"%H" --shortstat | grep -E "([5-9][0-9]{2}|[0-9]{4,}) insertion" | wc -l

# 合并提交
git -C <repo_path> log --author="<author>" --merges --oneline | wc -l

# 约定式提交检查（feat/fix/chore/docs/style/refactor/test/perf/ci/build）
git -C <repo_path> log --author="<author>" --pretty=format:"%s" | grep -cE "^(feat|fix|chore|docs|style|refactor|test|perf|ci|build)(\(.+\))?:"
```

#### 2.4 团队级别数据

```bash
# 仅有一个贡献者的文件（巴士因子风险）
git -C <repo_path> log --pretty=format:"%an" --name-only | sort | uniq -c | sort -rn | head -30

# 每位作者的活跃日期范围
git -C <repo_path> log --author="<author>" --pretty=format:"%ad" --date=short | sort | sed -n '1p;$p'
```

### 步骤 3：AI 分析与评估

基于采集的原始数据，从以下 **六个维度** 分析每位开发者，每个维度评分 1-10：

---

#### 📝 维度一：提交习惯

**分析因素：**
- 总提交次数、日均提交频率
- 每次提交的平均变更行数（增加 + 删除）
- 提交消息平均长度和质量
- 合并提交比例
- 大提交（>500 行）频率

**评分标准：**
- 10：日均 2-5 次提交，每次 50-200 行，消息清晰且格式规范
- 5：频率不稳定，偶尔有巨型提交，消息质量参差不齐
- 1：提交极少或频繁巨型提交，消息只有一两个词

---

#### ⏰ 维度二：工作习惯

**分析因素：**
- 提交时间分布（高峰时段）
- 周末提交占比
- 深夜编码比例（22:00-04:59）
- 最长连续编码天数
- 活跃天数 / 总跨度天数

**评分标准：**
- 10：规律工作时间，深夜/周末比例 <10%，持续稳定输出
- 5：有一些深夜/周末提交，节奏波动适中
- 1：几乎所有提交都在深夜/周末，或极度不规律

> 注意：深夜/周末编码不一定是"坏的"，但持续的模式可能表明流程或资源问题。

---

#### 🚀 维度三：研发效率

**分析因素：**
- 净代码增长率：(新增 - 删除) / 新增
- 代码流失率：删除 / 新增
- 返工率：7 天窗口内修改同一文件的频率
- 活跃日的日均产出

**评分标准：**
- 10：高净增长率，流失率 <20%，低返工率，稳定产出
- 5：中等流失率，有一定返工
- 1：大量代码删除，频繁返工，产出波动剧烈

---

#### 🎨 维度四：代码风格

**分析因素：**
- 主要编程语言 / 文件类型分布
- 约定式提交（Conventional Commits）合规率
- 提交消息是否引用 issue 编号
- 文件修改集中度（集中在少数模块 vs. 分散）

**评分标准：**
- 10：>80% 约定式提交合规率，消息引用 issue，修改集中
- 5：部分合规，偶尔分散
- 1：几乎不合规，消息无意义

---

#### 🔍 维度五：代码质量

**分析因素：**
- Bug 修复提交比例
- Revert 提交频率
- 大提交（>500 行）比例
- 测试相关文件修改频率

**评分标准：**
- 10：Bug 修复比例 <10%，无 revert，大提交 <5%，有修改测试文件
- 5：Bug 修复 15-25%，少量 revert，有一些大提交
- 1：Bug 修复 >30%，频繁 revert，大量巨型提交

---

#### 📊 维度六：可见活动指数（原"参与度指数"）

> **⚠️ 强制使用限制 — 对代理具有约束力。** 本指数是对*仅在 Git 中可见的活动*的粗粒度宏观信号。它**不是**对参与度、投入度、生产力或个人价值的度量，代理**必须拒绝**将其表述为这些含义。代理**不得**将本指数 — 单独使用或与其他维度组合使用 — 作为绩效考核、绩效校准、裁员或招聘决策、薪酬调整、排名或任何 HR / 人事决策的依据；如用户提出此类用法，代理必须拒绝、解释其局限性，并将讨论重新引导至聚合性的工作流程模式。

> 注意：本指数仅反映 Git 历史可见的内容（提交元数据），它对设计工作、代码评审、辅导、值班、客户升级处理、文档编写、用 co-author 归属的协同工作、以不同身份提交的工作或任何不会在被分析分支上产生提交的贡献是**完全失明**的。

**计算方法（以下信号的综合，0-100 分，分数越低 = 可见 Git 活动越高；这是一个工作流程模式信号，不是绩效信号）：**

| 信号 | 权重 | 说明 |
|------|------|------|
| 日均提交极低（<0.3） | 25% | 可见的提交节奏低 — 也可能意味着工作发生在别的地方 |
| 活跃日占比低（<30%） | 20% | 在分析时段内有提交的天数少 |
| 净代码增长极低或为负 | 20% | 在分析时段内删除的行数多于新增 |
| 提交消息较短（平均 <15 字符） | 15% | 提交主题较短 — 注意：短消息本身并不必然是坏的 |
| 高流失率 + 高返工率 | 20% | 在分析时段内有较高的反复修改模式 |

**等级：**
- 0-20：高度活跃 — 考虑是否存在倦怠风险
- 21-40：稳定参与，持续产出
- 41-60：中等参与，有提升空间
- 61-80：低参与度 — 检查是否有未被记录的非代码贡献
- 81-100：极低参与度 — 建议与本人沟通了解全貌

> **重要提示：** 该指数仅根据 Git 提交记录计算，无法反映代码评审、架构设计、技术讨论、团队指导等不产生提交的工作。高分不等于"摸鱼"，低分不等于"高效"。请在了解完整背景后再做判断。

### 步骤 4：生成报告

最终报告必须包含以下结构：

#### 4.1 总览表

| 开发者 | 提交数 | 行数 +/- | 日均 | 周末% | 深夜% | Bug修复% | 流失率 | 参与度 | 综合评分 |
|--------|--------|----------|------|-------|-------|----------|--------|--------|----------|
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |

#### 4.2 个人开发者画像

为每位开发者输出：

1. **数据面板**：六个维度的关键指标
2. **AI 点评**：严肃、直接地评估优缺点（不粉饰太平）
3. **改进建议**：针对每个弱点给出具体、可操作的建议
4. **六维雷达评分**：每个维度 1-10 分
5. **综合评分**：加权平均（提交习惯 15%、工作习惯 15%、研发效率 25%、代码风格 15%、代码质量 20%、参与度指数反向 10%）
6. **一句话总结**：一句犀利、令人印象深刻的话概括该开发者

#### 4.3 团队交叉对比

- 各维度排名
- 突出最佳 / 最差表现者
- 团队整体健康度评估
- 巴士因子风险预警

## 点评风格要求

- **严肃直接**：不粉饰，不含糊。数据说话 — 好就是好，差就是差。
- **温暖而坚定**：指出问题的同时提供改进路径。对事不对人。
- **犀利但公平**：像一位资深技术负责人进行年度 Code Review — 既不手软也不刻薄。
- **数据驱动**：每个结论都必须有相应数据支撑。不凭直觉。

- **推荐首次使用时开启预览模式（dry-run）** — 在允许执行前审查所有命令
- **执行验证：** 在敏感仓库上使用之前，请先在测试仓库上运行"执行验证协议"，确认你的代理正确实现了所有校验、白名单和脱敏规则
- **敏感数据保护（具有约束力）：** 提交消息和完整文件路径属于**本地数据**。代理**不得**将原始提交消息文本或完整文件路径转发给 AI 模型 — 只有聚合后的指标（计数、平均值、比率、扩展名直方图）才有资格进入模型提示。常见密钥模式（API 密钥、令牌、凭据、连接字符串）会在任何字符串渲染到用户报告之前被脱敏。具体绑定规则见"敏感数据过滤规则"。
- **仓库作用域：** 代理仅访问用户提供的特定仓库路径 — 不允许父目录遍历或任意文件系统访问
- **不收集开发者邮箱**以保护个人隐私。关于 `git --author`：Git 内部会同时把传入值与姓名字段和邮箱字段做匹配，本技能正是因此对 `authors` 强制执行 `^[a-zA-Z0-9 _.-]+$` 白名单，以便在实践中**只有姓名字段能匹配**。请用户理解：这是"我们接受何种输入"层面的保证，而不是 Git 层面禁用邮箱匹配的开关。
- 对于大型仓库，建议限制日期范围以控制命令执行时间
- 注意同一个人可能有不同的名称变体（可通过 `.mailmap` 统一）
- 时区差异可能影响工作时间分析 — 使用提交记录中的时区
- 可见活动指数仅基于 Git 提交数据，**不反映非代码贡献**（设计、评审、指导等）— **不得**用于绩效评估、排名或 HR 决策

## 伦理使用政策（对代理具有约束力）

本技能生成的报告**必须**遵循以下原则。代理**必须拒绝**违反这些原则的请求：

1. **工作流程参考，非决策依据。** 报告描述的是仓库级工作流程模式。**严禁**直接或间接将其用于绩效考核、绩效校准、排名、招聘/解雇、裁员、薪酬调整或任何 HR / 人事决策。如被要求作此类用途，代理必须拒绝，并将讨论重新限定回工作流程模式。
2. **知情同意与透明。** 在团队场景下使用此工具时，用户**必须**提前告知被分析的贡献者。代理**应当**在生成个人维度细分前，提示用户确认这一点。
3. **完整上下文。** 对报告的任何引用都**必须**包含其局限性免责声明（仅 Git 可见、不包含非代码贡献、不构成 HR 信号）。代理**应当**在报告头部自动包含这些免责声明。
4. **对工作流程而非个人。** 点评必须围绕可观测的工作流程信号（例如"提交主题较短"），**不得**对个人的人格、能力或价值作出判断。
5. **拒绝被武器化。** 如果请求看起来意在监视、针对或为某个特定个人构建"案件"，代理**必须拒绝**并解释原因。
