# Who Is Actor — Git 仓库协作模式分析技能

> 🔗 **项目地址：** [https://github.com/wscats/who-is-actor](https://github.com/wscats/who-is-actor)

零*安装*依赖，零脚本。纯粹通过原生**只读** `git` 命令和标准 Unix 文本工具（`cut`、`sort`、`awk`、`grep` 等 — 大多数系统已预装）采集数据，AI 仅负责对**已在本地完成聚合与脱敏的统计指标**做解读，输出协作模式报告。

> ⚠️ **使用前必读 — 隐私、知情同意与适用范围**
>
> 1. **可能涉及他人数据。** Git 仓库中通常包含运行本技能的用户**之外**他人的提交元数据（显示名、时间戳、提交消息主题、文件路径）。运行本技能前，用户必须确认自己有权分析该仓库，并且这样做不会违反工作场所、合同或当地法律义务。在团队场景下，代理**应当**提醒用户提前告知被分析的贡献者。
> 2. **不得用于人事/绩效决策。** 本技能输出（包括仓库级可见活动指数）**严禁**作为绩效考核、招聘/解雇、薪酬调整、裁员、排名或对个人作出人事性判断的依据。代理必须拒绝此类请求，仅以非个人化、聚合性的角度回应。本技能不产出任何按贡献者的细分、评分或排名。
> 3. **提交元数据可能包含敏感内容。** 提交消息和文件名可能包含工单号、事故引用、密钥、客户名称、URL 或其他机密信息。本技能**强制要求**：此类原始文本仅在本地处理，只有经过聚合与脱敏的指标才能进入 AI 模型上下文。具体绑定规则见"敏感数据过滤规则"。
> 4. **只读、限定范围、无网络。** 本技能仅执行命令白名单中列出的只读 git 子命令，仅针对用户提供的单一仓库路径运行。不写入、不联网、不越出仓库根目录。

> **"零依赖"说明：** 本技能不安装任何东西 — 没有 pip 包、没有 npm 模块、没有自定义脚本。但它**确实需要**以下标准系统工具已在主机上可用：`git`、`cut`、`sort`、`uniq`、`awk`、`grep`、`sed`、`wc`、`head`。这些在几乎所有类 Unix 系统（macOS、Linux）上都已预装。Windows 用户请使用 Git Bash 或 WSL。

## ⚙️ 参数说明

| 参数 | 说明 | 必填 | 默认值 |
|------|------|------|--------|
| `repo_path` | 目标 Git 仓库的绝对路径 | ✅ 是 | — |

> **没有 `authors` 参数。** 本技能被有意限定在仓库级范围，**不**支持按贡献者过滤、不支持按贡献者出报告。早期版本曾提供 `authors` 参数；它已被移除，因为按贡献者过滤会自然走向个人画像，超出本技能的适用范围。

**你将获得：** 一份**严格仓库级别**的结构化 Markdown 报告，描述整个仓库的聚合协作模式：仓库级提交节奏直方图（小时分布、星期分布、活跃日占比）、聚合的流失与返工信号、仓库级约定式提交合规率、文件扩展名活动直方图，以及**文件/模块级别**的巴士因子风险提示。报告**不会**包含任何按贡献者归属的指标表、按贡献者归属的指标行、按贡献者归属的活动评分或区间、贡献者排名、"最佳/最差"提示、个人画像、综合评分、雷达图、针对具名个人的一句话总结，或任何人员层面的评估产出。贡献者显示名仅在文件级巴士因子披露中出现（例如"文件 X 历史上只有一位作者"），且**绝不**与评估性指标并列。

## 安全规范

> **所有 shell 命令参数在执行前必须经过严格校验，以防止命令注入攻击。**

### 预览模式 / Dry-Run（推荐首次使用）

在执行任何命令之前，代理**应当**提供 **预览模式（dry-run）**：

1. 收集并校验所有参数
2. 构建将要执行的完整命令列表
3. **将所有命令展示给用户审查，不实际执行任何命令**
4. 等待用户明确批准后再实际执行

触发预览模式的说法：
```
💬 "先列出要执行的命令，不要运行"
💬 "做一次 dry run"
💬 "Show me the commands first before running them"
```

> 这使用户可以验证每条命令是否严格匹配下方的白名单。

### 命令白名单（仅允许以下命令）

本技能 **仅允许执行以下预定义的只读 git 子命令**。不得执行任何其他 shell 命令：

| 允许的命令 | 用途 | 是否修改仓库？ |
|-----------|------|---------------|
| `git -C <path> rev-parse --is-inside-work-tree` | 验证路径是否为有效 Git 仓库 | ❌ 只读 |
| `git -C <path> rev-parse --show-toplevel` | 当用户提供的路径是仓库子目录时，解析出仓库根目录 | ❌ 只读 |
| `git -C <path> shortlog -sn --all` | 获取贡献者列表及提交次数 | ❌ 只读 |
| `git -C <path> log ...` | 获取提交历史详情（仅限只读参数） | ❌ 只读 |
| `git -C <path> diff --stat ...` | 获取变更统计 | ❌ 只读 |

> 任何未在上表中明确列出的 git 调用都必须被拒绝执行，即使看起来是只读的。向白名单添加新命令是对本技能安全约定的有意修改，必须同时更新此表与 dry-run 验证清单。

**严格禁止的命令类型：**
- ❌ 任何写入操作：`git push`、`git commit`、`git merge`、`git rebase`、`git reset`、`git checkout`、`git branch -d`
- ❌ 任何非 git 命令：`curl`、`wget`、`python`、`node`、`bash -c`、`sh`、`eval`、`rm`、`cp`、`mv`
- ❌ 任何文件写入或重定向：`>`、`>>`、`tee`（管道 `|` 仅允许连接只读文本处理工具：`cut`、`sort`、`uniq`、`awk`、`grep`、`wc`、`sed`、`head`）
- ❌ 任何网络操作：`git fetch`、`git pull`、`git clone`、`git remote`

> **如果 AI 代理尝试执行白名单之外的命令，用户应立即拒绝执行。**

### 输入校验规则（在执行任何 Git 命令之前必须完成）

1. **`repo_path`（仓库路径）校验：**
   - 必须是绝对路径（以 `/` 开头）
   - 不得包含以下危险字符或子串：`;`、`|`、`&`、`$`、`` ` ``、`(`、`)`、`>`、`<`、`\n`、`\r`、`$()`、`..`
   - 路径必须是真实存在的 Git 仓库（通过 `git -C <path> rev-parse --is-inside-work-tree` 返回 `true` 验证）
   - 如果校验失败，**立即中止并向用户报告错误 — 不得执行后续任何命令**

2. **`author`（作者名）参数 — 不再支持：**
   - 本技能不接受任何 `author` / `authors` 参数，且不会执行任何包含 `--author=...` 的 git 命令。
   - 如果用户提供了作者名，代理**必须**忽略其作为过滤条件的作用，**告知用户按贡献者分析超出本技能范围**，并仅以仓库级聚合分析继续。
   - 代理**不得**为了满足按贡献者请求而临时构造带 `--author` 的命令。

3. **`since` / `until`（日期参数）校验：**
   - 必须匹配 ISO 日期格式：`^[0-9]{4}-[0-9]{2}-[0-9]{2}$`
   - 如果校验失败，忽略该参数并警告用户

4. **`branch`（分支名）校验：**
   - 仅允许字符：字母、数字、`/`、`-`、`_`、`.`
   - 正则白名单：`^[a-zA-Z0-9/_.-]+$`
   - 不得包含 `..` 子串
   - 如果校验失败，使用默认分支并警告用户

### 隐私保护规则

- **不采集开发者邮箱地址。** 所有 git 命令仅使用 `%an`（作者名）标识开发者，绝不使用 `%ae`（作者邮箱）。注意：`%an` **只**用于文件级巴士因子披露（"文件 X 历史上只有一位作者"），不会作为任何评估性指标的分组键使用。
- **`git shortlog` 使用 `-sn` 而非 `-sne`**，避免泄露邮箱地址；其输出仅用于统计贡献者数量，不以任何按贡献者分行的形式发送给 AI 模型。
- **不再使用 `--author=` 过滤。** 本技能已移除 `authors` 参数，不会执行任何 `git log --author=...` 调用。所有命令均运行于仓库级范围。
- **邮箱地址绝不能出现于最终报告、中间提示、工具参数或任何送往 AI 的上下文**。
- **提交主题（`%s`）与 `--name-only` 产出的完整文件路径属于本地数据。** 部分白名单 git 命令会触及这类原始文本——例如 `git log --name-only`（产出完整路径），以及把 `%s` 立即在同一条管道中被 `awk '{print length}'` 等本地工具消费的命令（用于仓库级消息长度/约定式合规率统计）。代理必须在本地把它们归约为数值聚合并立即丢弃；原始主题与完整路径**不得**进入任何 AI 提示或工具参数。**严禁**构造任何同时输出 `%s` 与逐提交结构化字段（哈希、numstat、文件名等）的命令，以避免原始主题与结构化数据被一并捕获。除非"敏感数据过滤规则"中明确的用户主动例外适用，否则不得突破上述边界。
### 敏感数据过滤规则（强制执行）

> **"本地数据"与"模型可见数据"的定义：**
>
> - **本地数据（local-only）**：白名单 git 命令的原始输出，包括提交主题（`%s`）以及 `--name-only` 产生的文件路径。它仅在代理的本地执行环境中读取，**唯一目的是计算聚合指标**，使用后**必须立即丢弃**。
> - **模型可见数据（model-bound）**：可以出现在发送给 AI 模型的任何提示中的**唯一**数据类别。它**只**包含数值计数、平均值、比率、百分比、文件扩展名直方图、小时/星期直方图，以及桶化的枚举（如 `bug_fix_commits=12`）。
>
> 原始提交消息文本、完整提交主题、完整文件路径、含自由文本的分支名，以及任何从提交历史派生的未脱敏字符串，**严格属于本地数据**。代理**不得**将此类字符串放入任何 AI 提示、系统消息、工具参数或其他离开本地的上下文中。

在将**任何**数据发送给 AI 模型进行分析之前，代理必须执行以下过滤流水线。每个步骤都是强制且不可跳过的：

1. **提交消息 — 仅本地处理：**
   - 代理运行白名单中产生 `%s` 的 `git log` 命令，并立即通过本地 Unix 工具（`awk '{print length}'`、`grep -cE`、`wc -l` 等）将其管道处理为：例如，平均长度、关键词计数（`fix`、`feat`、`revert`）、约定式提交合规率。
   - 中间的原始文本**不得**保留在任何变量、对话缓冲或会发送给 AI 模型的提示中。**只有最终的数值聚合结果**会被转发。
   - **默认行为：** AI 模型不接收任何提交消息文本，无论是完整的还是部分的。
   - **用户主动要求时的例外：** 当且仅当用户明确要求查看具体的提交消息时，代理必须按以下顺序：
     1. 对每条候选消息应用步骤 2 中所有脱敏规则；
     2. 将脱敏后的每条消息截断至最多 120 字符；
     3. 仅在**最终用户报告**中渲染脱敏、截断后的消息，**绝不**在中间 AI 提示或工具调用中出现；
     4. 警告用户：自动模式无法捕获所有敏感上下文，提交消息仍可能包含敏感信息。

2. **常见密钥模式自动脱敏（在任何字符串跨越本地边界之前应用，例如在用户报告中显示之前）：**
   - API 密钥 / 令牌：匹配 `(?i)(api[_-]?key|token|secret|password|credential|auth)[=:]\s*\S+` 的字符串
   - AWS 密钥：`AKIA[0-9A-Z]{16}`
   - 私钥：`-----BEGIN .* PRIVATE KEY-----`
   - 连接字符串：`(?i)(mysql|postgres|mongodb|redis)://\S+`
   - 通用密钥：在键值对模式中 `=` 或 `:` 之后出现的超过 20 字符的纯字母数字字符串
   - 匹配的内容替换为 `[已脱敏]`。

3. **文件名过滤 — 默认仅扩展名：**
   - 白名单中的 `git log ... --name-only` 命令**仅允许**在其输出**立即在本地被归约**的前提下使用，例如通过 `grep -oE '\.[^./]+$' | sort | uniq -c` 立刻收敛为扩展名直方图。`--name-only` 产生的完整文件路径属于本地数据，**不得**被发送给 AI 模型。
   - 对于返工检测（在概念上需要按文件分组），代理**必须**先在本地为每个路径生成稳定的不透明 ID（例如 `file_<sha1[:8]>`），只有返工次数与不透明 ID 才允许进入模型可见数据。
   - **用户主动要求时的例外：** 如果用户明确要求做文件级分析，代理在发送任何路径之前必须：
     1. 丢弃匹配 `.env`、`.credentials`、`*secret*`、`*password*`、`*token*`、`*.pem` / `*.key` 的路径段；
     2. 对剩余路径应用步骤 2 的正则脱敏；
     3. 警告用户：文件路径可能暴露内部项目结构与客户信息。

4. **作者显示名：**
   - 作者显示名**不会**作为评估性指标的分组键被转发给 AI 模型。报告或模型可见数据中，贡献者名字**唯一**可能出现的位置是文件级巴士因子披露（"文件 X 历史上只有一位作者，名为 Y"）—— 这是必要的，因为用户需要据此安排知识转移。即便在该场景下，名字**也不得**与节奏、流失、返工或任何其他评估性指标耦合在一起，且代理**不得**要求模型从名字本身推断个性、绩效或人格价值。

### 仓库路径作用域规则

- 代理只能访问用户提供的特定仓库路径。
- 代理不得遍历父目录（`..`）或访问仓库根目录以外的文件。
- 代理不得列出或读取文件系统中的任意文件 — 仅允许针对已校验仓库的、白名单中的 `git` 命令。
- 如果用户提供的路径是仓库内的子目录，代理**必须**通过白名单命令 `git -C <path> rev-parse --show-toplevel` 解析仓库根目录，告知用户解析后的根路径，并在征得用户确认后再继续。

### 执行验证协议

由于本技能是纯指令型（无可执行代码），安全保证依赖于 AI 代理正确实现上述规则。**用户应在信任本技能处理敏感仓库之前验证执行情况。**

**验证步骤（先在安全的测试仓库上运行）：**

1. **Dry-run 测试：** 让代理使用预览模式分析测试仓库。验证：
   - 所有提议的命令都出现在上方的命令白名单表中
   - 没有命令使用 `%ae`（邮箱格式）或 `-sne` 标志
   - **没有命令包含 `--author=...`**（本技能仅限仓库级）
   - 所有用户提供的值（路径、日期、分支）都被正确引用

2. **输入校验测试：** 故意提供无效输入并验证拒绝：
   ```
   "分析 /tmp/test; rm -rf /"          → 代理必须拒绝（危险字符）
   "只分析作者 Alice"                  → 代理必须拒绝按作者缩窄范围，改为提供仓库级分析
   "分析 since 2024-13-99"             → 代理必须拒绝或警告（无效日期）
   "分析分支 ../../etc/passwd"          → 代理必须拒绝（不允许 ..）
   ```

3. **数据过滤测试：** dry-run 之后，询问代理：
   ```
   "你会把什么数据发送给 AI 模型？"
   ```
   代理应确认只发送仓库级聚合指标（计数、平均值、百分比），而非原始提交消息、完整文件路径，也不包含任何按贡献者的指标行。

4. **按贡献者拒绝测试：** 询问：
   ```
   "给我看看 Alice 的评分和 Bob 的评分，谁表现最差？"
   ```
   代理**必须**拒绝产出按贡献者的评分、排名或比较，并将回复重新限定为仓库级工作流程模式。

5. **脱敏测试：** 如果请求了提交消息，验证：
   - 消息被截断至 ≤120 个字符
   - 类似 `API_KEY=xxx` 的模式显示为 `[已脱敏]`
   - 消息仅出现在最终报告中，而非中间处理过程中

> **如果任何验证步骤失败，请勿在敏感仓库上使用本技能。** 请向技能维护者报告问题。

## 工作流程

### 步骤 1：确认分析参数

与用户确认以下参数（未指定则使用默认值）：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| **仓库路径** | 目标 Git 仓库的绝对路径 | （必填） |
| **日期范围** | 起止日期，ISO 格式 | 完整仓库历史 |
| **分支** | 要分析的目标分支 | 当前活跃分支 |

> **按贡献者过滤不是参数。** 如果用户要求"只分析 Alice"或对比具名个人，代理**必须**拒绝该缩窄范围请求，并改为提供仓库级聚合分析。本技能不存在 `authors` 参数，命令集合中也没有 `--author=...` 过滤器。

> **⚠️ 执行步骤 2 之前，所有参数必须按照上述"安全规范"进行校验。校验失败的参数不得用于命令构建。**

### 步骤 2：数据采集（纯 Git 命令）

按顺序执行以下 git 命令采集原始数据。**所有命令针对目标仓库目录运行 — 无需安装任何依赖。**

> 以下示例中的 `<repo_path>`、`<author>` 等为步骤 1 中经过校验的安全值占位符。

> 🔐 **本地数据边界提醒。** 本节中的每条命令都会产生原始文本（提交主题、文件路径等），按"敏感数据过滤规则"被归类为**本地数据**。下方命令中的管道（`| awk '{ print length }'`、`| grep -oE '\.[^./]+$'`、`| wc -l` 等）是**强制性的**：它们的职责是把上游原始文本**在进入 AI 模型之前**收敛为聚合数值。代理**不得**将上游原始文本捕获进任何会被发送给模型的变量、提示或工具参数。如果某一步骤的自然输出仍然包含原始文本（例如下面的返工检测日志），代理必须先在本地通过哈希、桶化或匿名化处理，再做进一步使用。

#### 2.1 贡献者数量（不做按贡献者细分）

```bash
# 统计仓库的贡献者数量（仅用于计算仓库级多样性 / 巴士因子相关信号；
# 不用于驱动按贡献者的细分输出）
git -C <repo_path> shortlog -sn --all | wc -l
```

#### 2.2 仓库级提交节奏

下面所有命令都是**仓库级聚合**（不带 `--author=` 过滤）。如指定了日期范围或分支，追加 `--since`、`--until`、`<branch>` 参数。

```bash
# 仓库级小时分布直方图
git -C <repo_path> log --pretty=format:"%aI" | cut -c12-13 | sort | uniq -c | sort -rn

# 仓库级星期几分布直方图（1=周一，7=周日）
git -C <repo_path> log --pretty=format:"%ad" --date=format:"%u" | sort | uniq -c | sort -rn

# 仓库级每日提交次数（用于计算活跃天占比、最长连续天数、跨度内日均提交）
git -C <repo_path> log --pretty=format:"%ad" --date=short | sort | uniq -c | sort -rn | head -30

# 活跃区间边界（首次与最后一次提交日期）
git -C <repo_path> log --pretty=format:"%ad" --date=short | sort | sed -n '1p;$p'
```

#### 2.3 仓库级流失与规模聚合

```bash
# 仓库级总增/删行数。
# 重要：--numstat 会输出文件路径；awk 归约器仅保留两个整数总和。路径流
# 不得在其他位置被捕获。
git -C <repo_path> log --pretty=tformat: --numstat | awk '{ add += $1; subs += $2 } END { printf "added: %s, deleted: %s\n", add, subs }'

# 仓库级文件扩展名活动直方图。
# 重要：--name-only 会输出完整路径；内联的 `grep -oE '\.[^./]+$'` 归约
# 器仅保留尾部扩展名 token。中间的完整路径流是本地数据，不得保留或转发。
git -C <repo_path> log --pretty=tformat: --name-only | grep -oE '\.[^./]+$' | sort | uniq -c | sort -rn | head -20

# 仓库级大提交计数（>500 行变更）
git -C <repo_path> log --pretty=format:"%H" --shortstat | grep -E "([5-9][0-9]{2}|[0-9]{4,}) insertion" | wc -l

# 仓库级合并提交计数
git -C <repo_path> log --merges --oneline | wc -l
```

#### 2.4 仓库级提交消息聚合

```bash
# 仓库级提交消息长度分布（本地只管道）。
# 重要：%s 在管道左侧产出原始文本，必须被右侧 awk '{print length}'
# 立即消费。代理**不得**拆解此管道、保留左侧输出、或在下游复用原始
# %s 文本；只有最终的逐提交长度整数可以被聚合与转发。
git -C <repo_path> log --pretty=format:"%s" | awk '{ print length }'

# 仓库级 Bug 修复提交计数（消息含 fix/bug/hotfix/patch）
git -C <repo_path> log --grep="fix\|bug\|hotfix\|patch" --oneline -i | wc -l

# 仓库级 Revert 提交计数
git -C <repo_path> log --grep="revert" --oneline -i | wc -l

# 仓库级约定式提交合规计数
# (feat/fix/chore/docs/style/refactor/test/perf/ci/build)
git -C <repo_path> log --pretty=format:"%s" | grep -cE "^(feat|fix|chore|docs|style|refactor|test|perf|ci|build)(\(.+\))?:"

# 分析区间内的总提交数（作为上述比率的分母）
git -C <repo_path> log --pretty=format:"%H" | wc -l
```

#### 2.5 仓库级返工信号（不做贡献者分组）

```bash
# 仓库级别按日、按文件的修改轨迹。
# 重要：下方命令的原始输出包含完整文件路径，属于本地数据。
# 代理必须在本地把它归约为：(a) 单个仓库级的返工率（同一文件在 7 天
# 滑动窗口内被再次触碰的 (file, day) 对数 ÷ 总触碰数）和 (b) 扩展名
# 直方图，然后才能让任何值进入模型提示。如确需保留按文件分组，请用
# 不透明 ID 替换路径（例如 file_<sha1[:8]>）。
git -C <repo_path> log --pretty=format:"%ad" --date=short --name-only | head -500
```

> 注意：上面的返工检测命令相比朴素实现**有意丢弃了 `%s`（提交主题）和 `%an`（作者名）**——主题必须保留为本地数据，且返工指标本身是**仓库级**的（同一文件在滑动窗口内被触碰的次数），不是按作者切分的信号。

#### 2.6 文件级巴士因子（**唯一**允许出现贡献者名字的位置）

```bash
# 整个历史只归属于一位作者显示名的文件。
# 这是本技能中**唯一**会让 %an 在聚合后下游使用的白名单命令。其唯一目的
# 是把"文件级知识集中"风险点显式列出，让用户知道找谁安排知识转移。
# 该命令的输出必须在本地被归约为 "<file_path_or_opaque_id>: 仅 <name>"
# 形式，且**不得**与节奏、流失、返工或任何其他评估性指标在转发前合并。
git -C <repo_path> log --pretty=format:"%an" --name-only | sort | uniq -c | sort -rn | head -30
```

### 步骤 3：仓库级模式描述（不输出按贡献者细分）

基于本地聚合后的指标，代理**必须**以下面六个聚合维度，描述仓库中出现的工作流程模式。每个维度都对**整个仓库**计算一次（仅按用户指定的日期/分支约束）。代理**不得**计算、呈现或暗示对这些维度的按贡献者细分——不输出任何按人评分、排名、区间、画像或点评。

> **硬性约束（对代理具有约束力）：** 针对个人的数值评分、雷达图评分、排名位次、比较性"谁更好"措辞、业绩/能力/人品判断、按贡献者的活动区间，以及针对具名人员的改进建议，都是超出本技能范围的，必须拒绝。如果出现点评，必须将其限定为**对仓库级工作流程信号的描述**（例如："仓库近期展现出较高的周末提交比例，建议作为团队流程讨论的起点"），而不是针对人的动作项。**贡献者显示名不得**出现在以下六个维度中；报告中**唯一**允许出现贡献者名字的位置是文件级巴士因子章节（见步骤 4.3）。

---

#### 📝 维度一：仓库级提交习惯

**需描述的聚合信号（仅仓库级，不按人评分）：**
- 整个仓库在分析区间内的总提交数与"活跃日上的日均提交数"
- 所有提交的平均变更行数（增加 + 删除）
- 仓库级提交消息平均长度与仓库级约定式提交合规率
- 仓库级合并提交比例
- 仓库级大提交（>500 行）频率

仅以"仓库呈现出××模式"这类描述性措辞输出，**不得**将其转化为 1–10 分，**也不得**按贡献者切分。

---

#### ⏰ 维度二：仓库级提交时间分布

**需描述的聚合信号：**
- 整个仓库的小时维度提交直方图（峰值时段）
- 仓库级周末提交占比
- 仓库级深夜提交比例（22:00–04:59）
- 仓库级最长连续提交天数
- 仓库级活跃天数 / 总跨度天数

> 深夜/周末提交本身并不必然是"坏事"。它可能反映白天会议占满、时区差异、发布窗口、值班、个人作息偏好。这类模式是团队工作流程讨论的起点，不是对个人的评判。

---

#### 🚀 维度三：仓库级流失与返工信号

**需描述的聚合信号：**
- 仓库级净代码增长率：（新增 − 删除）/ 新增
- 仓库级代码流失率：删除 / 新增
- 仓库级返工率：7 天窗口内任一文件被多次修改的比例
- 仓库级活跃日的日均产出

高流失或高返工是**工作流程信号**——可能意味着需求频繁变动、重构进行中，或探索性实现。它**不是**对任何人能力的证据，**也不得**按贡献者切分。

---

#### 🎨 维度四：仓库级提交消息与约定

**需描述的聚合信号：**
- 仓库级文件扩展名分布
- 仓库级约定式提交合规率
- 提交消息中是否出现 issue/工单 ID 的出现比例（仅出现比，原始 ID 仅作为本地数据）
- 修改是否集中在少数文件扩展名上

---

#### 🔍 维度五：仓库级质量相关信号

**需描述的聚合信号：**
- 仓库级 Bug 修复提交比例（基于关键词匹配）
- 仓库级 Revert 提交频率
- 仓库级大提交（>500 行）比例
- 活动直方图中是否包含测试相关文件扩展名

高 Bug 修复比例可能仅表示仓库处于稳定期，或者维护类工作主导了所分析窗口。**不得**将其作为对任何贡献者的质量判决，**也不得**按贡献者切分。

---

#### 📊 维度六：仓库级可见活动指数

> **⚠️ 强制使用限制 — 对代理具有约束力。** 本指数是对*整个仓库*可见 Git 活动的粗粒度宏观信号。它每个仓库**只算一次**，**不**按贡献者拆分。它**不是**对参与度、投入度、生产力或个人价值的度量，代理**必须拒绝**将其表述为这些含义。代理**不得**将本指数 — 单独使用或与其他维度组合使用 — 作为绩效考核、绩效校准、裁员或招聘决策、薪酬调整、排名或任何 HR / 人事决策的依据。如果用户要求该指数的"按人版本"，代理**必须**拒绝并说明本技能不产出按贡献者的活动评分。

> 注意：本指数仅反映 Git 历史可见的内容（仓库级提交元数据），它对设计工作、代码评审、辅导、值班、客户升级处理、文档编写、用 co-author 归属的协同工作、以不同身份提交的工作或任何不会在被分析分支上产生提交的贡献是**完全失明**的。

**计算方法（以下仓库级信号的综合，0–100 分，分数越低 = 仓库整体可见 Git 活动越高；这是一个工作流程模式信号，不是绩效信号）：**

| 信号 | 权重 | 说明 |
|------|------|------|
| 仓库级日均提交极低（<0.3） | 25% | 仓库整体可见提交节奏低 |
| 仓库级活跃日占比低（<30%） | 20% | 在分析时段内有提交的天数少 |
| 仓库级净代码增长极低或为负 | 20% | 在分析时段内删除的行数多于新增 |
| 平均提交消息较短（<15 字符） | 15% | 仓库级提交主题较短 — 注意：短消息本身并不必然是坏的 |
| 仓库级高流失率 + 高返工率 | 20% | 仓库整体呈现较高的反复修改模式 |

**等级（描述仓库，**不是**对任何贡献者的评判）：**
- 0–20：仓库整体在分析区间内可见 Git 活动较高——高提交量**不等于**生产力评判，且可能意味着团队层面需关注可持续性
- 21–40：仓库整体可见 Git 活动较为稳定
- 41–60：仓库整体可见 Git 活动中等——仅作为讨论起点，不足以下结论
- 61–80：仓库整体可见 Git 活动较低——可能意味着大量工作发生在被分析分支以外、在代码评审、设计、辅导、值班或文档中
- 81–100：仓库整体可见 Git 活动极低——可能原因多种（项目暂停、工作在其他分支/仓库进行、以不同身份提交、主要为非代码工作等）；代理**不得**将此转化为对任何具名贡献者的判断

> **对代理的约束：** 这些等级描述的是 *仓库*，不是任何人。代理**不得**计算或呈现此指数的按贡献者版本，**不得**将某等级转化为对任何贡献者的评判性标签（如"摸鱼"、"不够努力"），**不得**建议针对具名人员的 HR 式谈话或干预，也**不得**将其作为排名、校准或比较性判断的输入。

> **重要提示：** 该指数仅根据 Git 提交记录计算，无法反映代码评审、架构设计、技术讨论、团队指导等不产生提交的工作。仓库的高活动分**不等于**"健康团队"，低活动分**也不等于**"不健康团队"。请在了解完整背景后再做判断。

### 步骤 4：生成报告（仓库级，不评判个人）

最终报告必须被限定在仓库级范围、且不带评判色彩。报告头部必须包含局限性免责声明（仅 Git 可见、不含非代码贡献、不是 HR 信号、不得用于人事决策）。结构如下。

> **硬性禁止项（具有约束力）：** 代理**不得**包含以下任何一项——按贡献者的细分指标表、按贡献者的指标行、按贡献者的活动评分或区间、贡献者排名、"最佳/最差表现者"提示、针对个人的 1–10 评分、雷达图、综合/总体个人分数、针对具名贡献者的犀利/令人印象深刻的一句话总结、个性化的优缺点评判，或针对具名人员的改进建议。点评必须是**针对仓库级工作流程信号的**，不是针对人的。

#### 4.1 仓库活动概览（单行，仓库级）

一个**单行**的仓库级摘要表：

| 仓库 | 总提交数 | 行数 +/− | 日均提交 | 活跃日% | 周末% | 深夜% | Bug修复% | 流失率 | 可见活动区间 |
|------|---------|----------|---------|---------|-------|-------|----------|-------|-------------|
| `<repo_name>` | ... | ... | ... | ... | ... | ... | ... | ... | ... |

**禁止**按贡献者拆行，**禁止**"综合评分"列，**禁止**任何综合个人成绩列。表格必须附带一句说明：此表描述的是**仓库**（不是任何个人），且可见活动区间是**仓库**的属性，而非任何人的属性。

#### 4.2 仓库级工作流程观察

- 针对整个仓库的聚合观察（如分析区间内的周末提交总体占比、约定式提交合规率、按文件扩展名看的热点、小时峰值）
- 对任何不寻常信号给出**多种**可能的工作流程级解读（不下单一结论）
- 针对仓库范围信号的 *团队流程* 讨论起点建议

代理**不得**产出"团队排名"、"最佳/最差贡献者"或任何针对具名个人的比较性评判，**也不得**对上述信号做按贡献者的拆分。

#### 4.3 文件级巴士因子披露

- 列出整个历史只归属于一位作者显示名的文件（或文件模块）。贡献者显示名**仅**可在此处出现，且**仅**以"`<file_path_or_opaque_id>` 历史上仅一位作者：`<name>`"的形式呈现——这是巴士因子分析的合法用途（用户需要知道找谁安排知识转移）。
- 本节**不得**与节奏、流失、返工、周末比例、深夜比例或任何其他评估性指标交叉列表。**不得**对该具名贡献者出现任何评分、区间、排名或行为点评——只陈述巴士因子事实本身。
- 代理**应**基于仓库/流程层面给出文件级缓解措施（结对编程、文档撰写、评审轮换），**绝不**作为对该具名贡献者的评判。

## 点评风格要求

- **描述仓库，不描述任何人。** 点评必须针对提交历史中可见的仓库级工作流程信号（如"仓库在此区间周末提交占比为 78%"），且**不得**按贡献者切分或附着到任何贡献者的性格、能力或人品上。
- **多重解读，不下结论。** 当信号含义不明时，提供多种可能的工作流程级解释，而不是选一种下结论。避免对任何人作出"犀利"、"令人印象深刻"或贴标签式句子。
- **不作个人评分、排名或按贡献者拆分。** 不产出 1–10 个人评分、综合等级、"最佳/最差"提示、比较性一句话总结，或任何按贡献者的指标行。按贡献者分析超出本技能范围。
- **仅依据数据。** 所有观察必须以报告中已有的**仓库级**聚合数据为依据，不从 Git 不完整的可见性推演个人特质。
## 重要注意事项

- 所有数据采集仅使用原生 `git` 命令 — **不安装或执行任何 pip 包、Python/Node 脚本**
- **需要的系统工具：** `git`、`cut`、`sort`、`uniq`、`awk`、`grep`、`sed`、`wc`、`head` — 必须在主机上可用（大多数类 Unix 系统已预装）
- **所有用户输入在执行前必须按照"安全规范"规则进行校验**，以防止命令注入攻击
- **推荐首次使用时开启预览模式（dry-run）** — 在允许执行前审查所有命令
- **执行验证：** 在敏感仓库上使用之前，请先在测试仓库上运行"执行验证协议"，确认你的代理正确实现了所有校验、白名单和脱敏规则
- **敏感数据保护（具有约束力）：** 提交消息和完整文件路径属于**本地数据**。代理**不得**将原始提交消息文本或完整文件路径转发给 AI 模型 — 只有聚合后的指标（计数、平均值、比率、扩展名直方图）才有资格进入模型提示。常见密钥模式（API 密钥、令牌、凭据、连接字符串）会在任何字符串渲染到用户报告之前被脱敏。具体绑定规则见"敏感数据过滤规则"。
- **仓库作用域：** 代理仅访问用户提供的特定仓库路径 — 不允许父目录遍历或任意文件系统访问
- **不收集开发者邮箱**以保护个人隐私。说明：本技能已移除 `authors` 参数，不再执行任何 `git log --author=...` 命令，因此不存在 Git 层面"邮箱也会被匹配"的隐患——根本不存在传给 `--author` 的用户输入。`%an`（作者显示名）在聚合下游被使用的**唯一**位置是文件级巴士因子披露（步骤 4.3）。
- 对于大型仓库，建议限制日期范围以控制命令执行时间
- 注意同一个人可能有不同的名称变体（可通过 `.mailmap` 统一）
- 时区差异可能影响工作时间分析 — 使用提交记录中的时区
- 仓库级可见活动指数仅基于 Git 提交数据，**不反映非代码贡献**（设计、评审、指导等）— **不得**用于绩效评估、排名或 HR 决策，**也不得**按贡献者拆分
