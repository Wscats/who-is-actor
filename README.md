代码仓库里藏着一个团队的协作节奏：什么时间提交、多频繁、多集中、有多少返工。<br>
本技能把这些**仓库级模式**汇成一份团队可以共同阅读与讨论的报告。

**它是什么 / 不是什么？** 一个轻量、零安装的**仓库级协作模式**镜子 —— **不是**开发者画像工具、**不是**报告卡、**不是**绩效评分、**不是**人事工具。报告中出现的姓名仅用于把指标归属到 Git 提交记录，不构成对个人能力或价值的判断。

<br>

零依赖 · 零脚本 · 只读原生 `git` 命令<br>
不装 pip，不装 npm，不跑 Docker

## ⚠️ 隐私与适用范围（必读）

在打开这个技能之前，请先看完这几点：

1. **这不是 HR / 绩效 / 个人画像工具。** 本技能输出的是**仓库级**协作模式报告。其中的任何分数、"可见活动指数"以及按贡献者归属的细分指标，**严禁**用于绩效考核、招聘/解雇、裁员、薪酬调整、排名、个人能力评判或任何其他人事性判断。它是代码仓库使用习惯的镜子，**不是**个人画像、**不是**对人的评分。Git 看不到设计、代码评审、辅导、值班、文档以及各种不产生 commit 的贡献。
2. **仓库里会出现别人的名字。** Git 提交元数据包含贡献者的显示名、提交时间、文件路径、提交主题等。运行本技能前，**请确认你有权限分析这个仓库，且这样做不会违反工作场所、合同或当地法律义务**。在团队场景下使用时，**应提前告知被分析的同事**。
3. **明确告知会被发送给 AI 模型的内容（重要）。**
   - **会发送给 AI 的，仅限本地聚合后的数值指标**：计数、平均值、比率、百分比、文件**扩展名**直方图、小时/星期直方图，以及类似 `bug_fix_commits=12` 的桶化枚举。
   - **不会发送给 AI 的本地数据**：原始提交消息文本（`%s`）、完整文件路径（来自 `--name-only`）、含自由文本的分支名、任何未脱敏的字符串。这些数据仅在本地内存中作为中间值用于计算上述指标，使用后立即丢弃。
   - **唯一例外**：用户主动要求查看具体提交消息时，代理必须先做密钥脱敏、再截断到 ≤120 字符，且**仅出现在最终用户报告中**，绝不进入中间 AI 提示。
   - 即便如此，提交主题和文件名仍可能在采集瞬间出现工单号、事故号、客户名、内部 URL、不小心提交的密钥等。**如果仓库可能含高敏感信息，请先在本地复制一份并谨慎使用。**
4. **只读、限范围、不联网。** 本技能仅调用只读 git 白名单命令，不写入、不拉取/推送、不越出仓库根目录。首次使用请开启 Dry-Run 预览模式。

# 仓库可见的协作模式，能说明一些什么

开发团队的工作，有很大一部分是 Git 看不见的：设计讨论、代码评审、辅导、值班、文档、踩坑备忘录。
Git 能看见的那一部分，也未必是完整的故事。

但仓库提交历史仍然包含一些有用的信号：提交节奏、文件热点、返工频率、巴士因子风险等。本技能把这些信号汇成一份**仓库级**报告，给团队一个共同阅读与讨论的起点。

它不会告诉你谁努力、谁不努力。它只会告诉你：这个仓库的提交发生在什么时间、多集中、多少被重写。**先别急着下结论。**
# 怎么用

作为 Skill 插件使用。**触发激活的标准是显式且无歧义的**——请在请求中**同时包含**：

1. **明确的"分析这个 Git 仓库"意图**——单独说"分析一下研发效率 / 工作习惯 / 代码质量 / 团队参与度"等泛泛话题**不足以**激活本技能；
2. **具体的仓库绝对路径**（例如 `/Users/me/code/my-repo`），缺少路径时代理必须先向你索取；
3. 默认采用 Dry-Run 预览模式，先看要跑哪些命令，你同意了再跑；
4. 在团队仓库上运行前，确认你有权分析该仓库，**并已告知被分析的同事**。

示例（最推荐）：

```
“请对 /Users/me/code/my-repo 这个 Git 仓库运行 who-is-actor，先 Dry-Run。”
“在 /Users/me/code/my-repo 上生成仓库协作模式报告。”
```

> 当你只说出"分析一下这个仓库的研发效率""看看团队的参与度""比较一下两位同事的代码质量"这类泛泛描述、或试图让代理对具体个人作出评判时，**代理不应自动开始采集**，而必须先与你确认：(1) 具体的仓库路径，(2) 你是否有权限分析，(3) 团队成员是否已被告知，(4) 是否使用 Dry-Run。任何针对个人的"对比 / 排名 / 评判"诉求都超出本技能范围，代理应拒绝并改为提供仓库级工作流程模式描述。

## 把一个 Git 仓库交给他

本技能负责**一件事**：从仓库提交历史中，提炼各种可量化的协作模式、写代码的节奏、周末/深夜比例、代码流失、文件反复修改、巴士因子风险等。

这是一份面向 *团队回顾与流程改进* 的参考资料，而不是人事报告。

零依赖，零脚本，只用只读 `git` 命令。

## 它能看到什么

六个**仓库级模式**维度。所有指标均为聚合后的可见表现，请结合实际上下文阅读，不要单点下结论。

### ⏰ 提交时间分布

一天内提交集中在哪些时段？周末的提交占多少？最长连续提交跨几天？
这些表现只告诉你**什么时间仓库被写入**，不告诉你**为什么**。
有人周末提交是加班，有人只是周末才有静下心思考的时间。

### 📉 代码流失与净增长

在分析区间内，新增与删除的比例。高流失率可能是反复重写，也可能是需求变动，还可能是在重构。
这个指标适合引发讨论，不适合下判词。

### 📝 Commit message 的长度与约定式提交合规率

平均多长？是否遵循 `feat:` / `fix:` 此类约定。  
出于隐私原因，原始消息文本**不会**被发送给 AI 模型，只会发送长度、关键词计数、合规率这类数值。如需查看原始消息，需你主动指明，并会经脱敏后仅在报告中出现。

### 🔄 同一文件的返工频次（Rework Ratio）

7 天窗口内同一文件被多次修改的频率。高返工可能说明：变更频繁的需求、迭代运行中的模块、或者必须提前介入设计论证的部分。

### 🐛 Bug fix 与 revert 类提交占比

仅供参考，不能以此判断个人质量。高比例可能说明该贡献者主要负责修复者角色，也可能说明近期高发事故。

### 📊 可见活动指数（Visible-Activity Index）

代替原来的“参与度指数”说法，强调它只映射 Git 里能看见的那一部分。

> **重要提醒：** 本指数**不是**参与度、投入度、生产力的度量，**不得**被用于人事决策。Git 不会记录设计、评审、文档、指导、值班、客户调研等不产生 commit 的贡献。

## 报告中你可能看到的模式（示例）

> 以下是为了让你明白报告长什么样子而列出的**示意性模式**。报告本身描述的是仓库中出现的**提交模式**，而不会为具体个人贴上“深夜战士”/“Bug 制造机”之类的人设标签。阅读这些示例时，请始终记住：同一组指标背后可能是你不知道的上下文。

### 🌙 深夜/周末提交集中的模式

```
Peak Hour:        23:00
Weekend Ratio:    78.6%
Churn Rate:       43.7%
```

提交集中在深夜与周末，伴随较高的流失率。
可能的所有读法均成立：白天被会议填满、项目迭代期本身偏周末、个人作息偏晚、重构阶段多次调整。
适合用来问“我们呢”，不适合用来说“他怎么了”。

---

### 🏗️ 集中的大提交 + 低流失率的模式

```
Total Lines Added:  84,224
Large Commit Ratio: 50.0%
Ownership Ratio:    100.0%
Churn Rate:         0.0%
```

仓库中出现一次性大范围提交、后续几乎不走动的区域，常见于初始化提交、主干迁入、仅一人维护的模块。
需要点出的是**巴士因子风险**：Ownership 100% 意味着该区域只有一位贡献者，一旦他不在，该部分代码会失去熟悉者。这是需要团队补一补知识转移的信号。

---

### 🎯 高周末比例 + 低流失率 + 低返工率的模式

```
Weekend Ratio:  66.7%
Churn Rate:     3.8%
Rework Ratio:   7.7%
```

提交集中在周末，但改动稳定、置换少。仓库中这类模式可能对应于：设计阶段在工作日完成、实现阶段在周末集中落地；也可能对应于个人作息偏好。这些仅仅是假设，不是结论。

---

### 👻 低频但不可忽视的贡献

```
Total Commits:  1
Active Span:    1 day
```

只出现过一次的贡献者。Git 能告诉你提交只有一次，但告诉不了你这次提交背后是十分钟还是三周。报告不会也不应该从提交数量推断“贡献价值”。

---

### 🔥 高占比 Bug Fix 提交的模式

```
Bug Fix Commits:    47
Fix Rate:           80.9%
```

仓库近期存在大量以 fix/bug/hotfix 开头的提交。可能的原因包括：该阶段项目进入 bug bash / 上线前稳定期；某位贡献者主要负责修复者角色；或近期高发事故。**不能**单凭该指标判断质量，需结合近期事故记录与仓库阶段。

---

### 😴 低频低量的提交序列

```
Total Commits:      23
Last Active:        6 months ago
Avg Commit Size:    2 lines
```

近半年未有提交，历史提交均为小幅。可能的原因多样：贡献者转岗、主力转代码评审/需求、项目进入维护期。报告只会描述“该贡献者近期未有提交”这个事实，不会推断“该人状态”。

## 怎么做到的

整个流程就一条流水线，简单清晰：

```
用户显式请求 (含仓库路径) → 参数白名单校验 → (推荐) Dry-Run 预览
  → 白名单内的只读 git 命令采集 (产物为本地数据)
  → 本地脱敏 + 聚合 (提交主题/文件路径在此完成数值化)
  → 只把聚合后的数值指标喂给 AI 模型 → 生成仓库级协作模式报告
```

**关键边界（与隐私一致）：**
- **AI 模型只能看到**：计数、平均值、比率、百分比、文件扩展名直方图、小时/星期直方图、桶化枚举。
- **AI 模型看不到**：原始提交主题、完整文件路径、未脱敏的字符串、邮箱。
- 数据采集仅依赖白名单内的 git 只读命令（包括 `rev-parse`、`shortlog -sn`、`log`、`diff --stat`），不会联网、不会写入、不会越出仓库。

没有 Python，没有 Node，没有任何额外依赖。

## 常见问题

**Q：这个工具会不会让我被 leader 针对？**  
A：本技能严禁作为人事/绩效依据。如果被这样使用，请告知使用者其使用超出了本技能的伦理使用政策。报告描述的是仓库级协作模式，不是个人能力。

**Q：我的 commit message 都是 "fix"，会不会被骂？**  
A：AI 不会对个人作出贴标签式点评。报告只会描述“本仓库提交消息平均长度较短”这类仓库级事实。

**Q：我只有一个 commit，是不是很丢人？**  
A：Git 只看得到 commit，看不到他背后的工作量。报告不负责评价人。

**Q：我的流失率 80%，正常吗？**  
A：高流失率是个信号，不是结论。可能是需求变动、重构、迭代探索。合适的下一步是把它带去评审讨论，而不是被报告“定罪”。

**Q：这个工具会把我的 commit message 传到云端吗？**  
A：默认不会。原始提交消息（`%s`）和完整文件路径属于**本地数据**，仅在你本地的代理进程内被读入，立即被聚合为统计指标（平均长度、关键词计数、合规率、扩展名直方图）后丢弃。**进入 AI 模型上下文的只有这些数值**。唯一例外：你**主动要求**查看具体提交消息时，代理会先做密钥脱敏、截断到 120 字符，仅在最终报告里展示。

**Q：这个工具能分析我自己吗？**  
A：当然可以。在你自己的仓库上进行自我回顾，是本技能的推荐场景。

## 数据是镜子，不是法官

深夜提交不等于高效，也不等于不努力。  
高流失率不等于勤奋，也不等于混乱。  
大量 bug fix 不等于责任心强，也不等于能力不足。  
大量 commit 不等于高产，少量 commit 也不等于不负责。

每个指标背后都可能有一个你不知道的上下文：

- 那个凌晨三点提交代码的同事，也许是白天被会议填满的。
- 那个 commit message 只写 "fix" 的人，也许正在争分夺秒修复线上事故。
- 那个只贡献了一个提交的人，也许是花了整整一周读懂代码才动的手。
- 那个流失率 80% 的人，也许面对的是不断变化的需求。

代码会说话，但只有用心听，才能听到正确的答案。

本技能负责拿出镜子，不负责代你判人。
